## ❓REST API에 대해 설명해보세요. 
REST API는 REST 원칙을 지켜 설계한 웹 API입니다.
REST 원칙은 자원을 이름으로 구분하고 자원의 상태를 주고받는 아키텍처 스타일입니다. 
즉, 자원은 URL로 표현하고 행위는 HTTP 메서드로 구분하는 방식!

**➕ REST의 특징**
- 서버/클라이언트 구조 : 자원이 있는 쪽이 서버, 요청하는 쪽이 클라이언트로 역할을 구분
- 무상태 Stateless : 서버가 클라이언트의 상태를 기억하지 않고, 클라이언트는 요청할 때 모든 정보를 담고 있음
- 캐시 처리 가능 : 자주 쓰는 데이터를 매번 DB를 보는 대신 저장해두고 다음 요청 때 빠르게 꺼내쓸 수 있음
- 계층 구조 : Rest Server는 보안, 암호, 로드밸런싱을 위한 계층을 추가할 수 있고 Proxy, Gateway와 같은 중간매체를 사용할 수도 있지만 클라이언트는 서버와 직접 통신하는지 중간 서버와 하는지 알 수 없음
- 인터페이스 일관성 : 모든 API가 동일한 규칙을 따르도록 하는 원칙 즉,  HTTP 프로토콜에 따르는 모든 플랫폼에서 사용가능하며 언어나 기술이 종속되지 않음
- 자체 표현 : 요청 메시지만 보고도 쉽게 이해할 수 있는 자체 표현 구조로 되어있음
</br>

## ❓HTTP 상태 코드에 대해 설명해주세요.
HTTP 상태 코드는 클라이언트가 보낸 HTTP 요청에 대한 서버의 응답 코드로 상태 코드에 따라 요청의 성공이나 실패 여부를 판단할 수 있습니다.

- 100번대에는 요청이 수신되어 처리 중을 뜻하는데, 웹 개발에서 직접 사용할 일은 거의 없습니다.
- 200번대는 클라이언트 요청이 서버에서 성공적으로 처리되었음을 뜻하고,
- 300번대는 클라이언트가 요청한 리소스가 다른 위치에 있거나 재요청이 필요함을 나타냅니다.
- 400, 500번대는 둘 다 실패를 의미하는데, 400번대는 그 원인이 클라이언트 측에 있고 500번대는 서버 측에 있는 것을 의미합니다.

**➕ 401(Unauthorized)와 403(Forbidden)은 의미적으로 어떤 차이가 있나요?** 
- 401은 인증이 실패된 것이고 (즉 로그인이 불가하거나 토큰이 만료된 경우) 
- 403은 인증은 되었지만 권한이 없는 것입니다. (로그인은 되었지만 관리자 리소스에 접근은 안될 경우)

**➕ 200(ok)와 201(created)의 차이에 대해 설명해 주세요** 
- 200은 조회나 수정과 같은 요청에 대한 성공을 의미하고 (get,patch/put)
- 201은 요청으로 인해서 새로운 리소스가 생성되었다는 것을 의미합니다. (post)
</br>

## ❓HTTP Method에 대해 설명해 주세요.
HTTP Method는 클라이언트가 서버에 “무엇을 할지”를 명확하게 표현하는 약속입니다. </br>
즉 리소스에 대한 행위를 뜻합니다. → REST API에서 URI는 명사, Method는 동사 역할

**➕ HTTP Method의 멱등성에 대해 설명해 주세요.** </br>
한 번 호출하든 두 번 호출하든 100번 호출하든 결과가 같은 것입니다. </br>
단순히 돌아온 값이 같을 뿐 아니라 DB에도 영향을 미치지 않습니다.
- GET, PUT, DELETE는 멱등 메서드지만 POST, PATCH는 아님
- PATCH가 아닌 경우 : `PATCH /users/1  { "loginCount": "+1" }`

**➕ GET과 POST의 차이는 무엇인가요?** </br>
GET은 조회용으로 멱등성을 가지고 캐시가 가능하지만, POST는 서버 상태를 변경하기에 멱등하지 않습니다. </br>
또한 GET은 파라미터 위치가 URL에 있고 POST는 Body에 있습니다.

**➕ POST와 PUT, PATCH의 차이는 무엇인가요?**
- POST는 새로운 리소스를 생성하는 것이고
- PUT은 리소스 전체를 수정하는거라 요청을 보낼때 전체 필드를 보내고
- PATH는 리소스 일부만 수정하기에 변경이 필요한 필드만 전송하면 됩니다.

**➕ HTTP 1.1 이후로, GET에도 Body에 데이터를 실을 수 있게 되었습니다. 그럼에도 불구하고 왜 아직도 이런 방식을 지양하는 것일까요?**
1. HTTP 설계 원칙에 따르면 GET은 조회 역할이고 POST는 데이터 생성 역할로 분리되는데, Body를 포함한 GET 요청은 이러한 역할 분리를 어렵게 만듭니다.
2. 캐싱이나 프록시 설계 일부는 GET의 Body를 참조하지 않는 경우가 있어 호환성의 문제가 발생합니다.
